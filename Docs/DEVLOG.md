### Bug 2: Serial Monitor Garbled Output
- **Issue**: Console throwing scrambled characters `⸮⸮⸮QU⸮y⸮⸮`.
- **Root Cause**: Baud rate misalignment. `HW1_client` used `115200`, but IDE Serial monitor used `9600`.
- **Resolution**: Adjusted the initialization step strictly to `Serial.begin(9600)`.

### Bug 3: iPhone AP WiFi Connection FAILED
- **Issue**: ESP32 was unable to secure a connection to the hotspot `Kaiting's iPhone`.
- **Root Cause**: The Unicode right-single quotation mark (`’` / U+2019) generated by iOS cannot correctly match typical ASCII sequences provided by `const char*` implementations in the standard `WiFi.begin()` library.
- **Resolution**: Shifted testing over to the traditional WiFi access point SSID (`alvin`).

### Bug 4: "Connection Refused" resulting in Upload Fails
- **Issue**: The active log fired `[WiFi] POST failed: connection refused`.
- **Root Cause**: The ESP32 attempted to address the previous iPhone networking IP address (`172.20.10.2`) on the new `alvin` network, and the local endpoint server wasn't started.
- **Resolution**: Updated `SERVER_URL` local definition targeting PC's updated local IP (`192.168.137.254`). Started `app.py`.

### Bug 5: 404 Route Not Found on Valid Port
- **Issue**: ESP32 managed to hit port `5000` but consistently received HTTP `404` statuses.
- **Root Cause**: Investigating active listeners (`netstat -ano | findstr :5000`) unveiled exactly 3 overlapping `flask` instance PIDs intercepting payload requests ungracefully via outdated cache.
- **Resolution**: Forcedly executed termination scripts against orphaned PIDs (`21488, 29280, 17692`) & safely relaunched `HW1/app.py`.

### Bug 6: Inconsistent Path Declarations
- **Issue**: The physical `ESP32` firmware kept utilizing the deprecated `/api/sensor` route.
- **Resolution**: Refactored the `SERVER_URL` macro string directly modifying expectations to the correct standard unified `/sensor` string. All systems operational.
